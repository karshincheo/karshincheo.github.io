My [kalshi-trading](https://github.com/karshincheo/kalshi-trading) repo contains the thing I'm most proud of building this year: a research loop where an LLM writes trading strategies, gets graded by a backtester it can't touch, and commits its own improvements to git overnight.

This post is the architecture tour — including the parts that didn't work and the security claims I won't make.

## The problem

Kalshi runs daily temperature prediction markets: "Will the high in NYC today exceed 78°F?" These markets are small, weather is genuinely forecastable, and the gap between a weather model and a market price is a measurable edge. The catch: finding a good mapping from *forecast features* to *calibrated probability* is a grind of trying rules, backtesting, and tweaking. Exactly the kind of grind an LLM can do while I sleep.

## The loop

```
Collect Kalshi data → Build dataset (market snapshots + weather)
     ↓
Evaluate current strategy → Prompt Claude with code + metrics
     ↓
Extract code → AST validation → Sandboxed execution
     ↓
Score (Brier + Sharpe) → Accept/reject → Git commit
     ↓
Repeat
```

Five design decisions matter more than the rest:

**1. The optimization target is a calibration score, not profit.** Each iteration is judged on [Brier score](https://en.wikipedia.org/wiki/Brier_score) — mean squared error between predicted probabilities and what actually happened — on validation data the model never saw. Optimizing raw P&L overfits hilariously fast; a strategy that lucked into three good trades looks like a genius. Calibration is harder to game, and a Sharpe gate sits behind it as a sanity check.

**2. The agent never sees holdout data.** The dataset splits by `target_date` into train/validation/holdout. Claude gets feedback from validation; holdout is evaluated manually, once, at the end. This is the same discipline you'd want from a human quant, enforced by file paths.

**3. Generated code goes through two gates, and I'll only call one of them "security."** First, an AST pass: import whitelist, blocked builtins (`eval`, `exec`, `open`), no network modules. Second, the code runs in a separate subprocess with a 60-second timeout and a stripped environment. Here's the honest part: *the AST pass is a pre-filter, not a security boundary.* Classic escapes like `().__class__.__bases__[0].__subclasses__()` walk straight past a static import check — the test suite [pins this limitation explicitly](https://github.com/karshincheo/kalshi-trading/blob/main/backend/app/tests/unit/test_sandbox_validator.py) so it's a documented trade-off, not an accident. The subprocess isolation is what actually bounds misbehavior, and for hostile (rather than merely buggy) code you'd want a real jail — containers, seccomp, no shared interpreter. For code generated by a model I'm prompting myself, defense-in-depth at this level is the right cost/benefit. Claiming more would be marketing.

**4. Git is the lab notebook.** Every accepted strategy is committed with its scores in the message. The model's research history is a `git log` — diffable, revertable, auditable. When iteration 47 beats iteration 12, you can read exactly what changed in the strategy code between them.

**5. The evaluator is immutable.** The model can rewrite the strategy; it cannot rewrite the thing that grades the strategy. If the LLM can touch the eval, the eval is decoration.

## Run it yourself, no keys

The repo bundles a real collected dataset — 185 temperature-market snapshots with weather features — so the evaluator runs offline:

```bash
git clone https://github.com/karshincheo/kalshi-trading
cd kalshi-trading/backend && pip install -e .
make demo
# Brier Score: 0.0828 on the bundled sample, ~25s, zero API keys
```

That 0.0828 is the seed strategy (a conservative forecast-vs-strike rule) on bundled sample data — a demo of the measurement machinery, not an out-of-sample performance claim. The difference between those two sentences is most of what I learned building this.

## What was hard, honestly

- **Data is the real bottleneck.** The LLM loop was the fun weekend. The collector that polls market snapshots every five minutes, joins them to weather observations, and labels outcomes was the slog — and the loop is useless without it.
- **The model loves complexity.** Left alone, Claude reaches for ensembles and rolling-window meta-features. Most accepted improvements were boring: better handling of forecast lead time, saner clipping. The prompt now actively rewards simplicity.
- **Backtest plumbing leaks.** My first evaluator accidentally let strategies see same-day snapshots ordered after the prediction timestamp. Nothing teaches you look-ahead bias like an LLM exploiting it within four iterations.

## Where it goes

The loop currently optimizes one market family (daily temperature highs). The architecture generalizes to anything with a collectable feature set and a binary settlement — the strategy interface is one `predict(row) -> probability` function. Phase 2 is paper-trading the best strategy against the live feed; live capital, if ever, comes capped and last.
